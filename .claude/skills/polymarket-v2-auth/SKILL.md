# Polymarket V2 Authentication — Definitive Guide

**Created 05.05.2026** after multi-hour research into V2 auth model + live GitHub issues + a field report from a similar project. Read this BEFORE writing any V2 trading code. Most "INVALID_SIGNATURE" / "order_version_mismatch" / "401" / "maker address not allowed" failures trace to ONE of four mistakes documented here.

## When to use

- Initializing any V2 ClobClient (Python or TS)
- Diagnosing 400/401/403 from `clob.polymarket.com`
- Onboarding a new wallet for trading
- Writing helper scripts that derive API creds (`poly_derive_api_creds.py`)

For production order signing details see `polymarket-v2-connector`. For failure-mode matrix see `polymarket-v2-troubleshoot`.

---

## TL;DR — the only three auth combinations that exist

| Wallet path | `signature_type` | `signer` (key controls) | `funder` (holds pUSD) |
|---|---|---|---|
| **A. Pure EOA** — Metamask key created off-Polymarket, you funded the EOA directly | `0` (EOA) | your Metamask address | **same** as signer |
| **B. Magic email** — signed up via email, "Export Private Key" in UI (Magic.link) | `1` (POLY_PROXY) | Magic-derived EOA address | **proxy** address shown at polymarket.com/settings (a contract, not an EOA) |
| **C. Browser wallet on polymarket.com** — connected Metamask via UI; UI created Gnosis Safe proxy | `2` (POLY_GNOSIS_SAFE) | your Metamask address | **Gnosis Safe** address shown at polymarket.com/settings (a contract, not an EOA) |

**There is NO `signature_type=3`. There is NO `NONE`.** Only `0`, `1`, `2`. If you saw "3" somewhere — it's enum length, a typo, or wrong source. Confirmed against `py-clob-client-v2` v1.0.0 source and docs.polymarket.com/api-reference/authentication (verified 05.05.2026).

## Decision tree: which `signature_type` do I use?

```
Did you sign up by clicking "Connect Wallet" → Metamask on polymarket.com?
├─ YES → signature_type = 2 (POLY_GNOSIS_SAFE)
│         funder = address shown at polymarket.com/settings
│                  (MUST be a contract — not the deposit address!)
│         signer = your Metamask address
│         private_key = your Metamask private key
│
└─ NO. Did you sign up via email (Google/email) on polymarket.com?
    ├─ YES → signature_type = 1 (POLY_PROXY)
    │         funder = address shown at polymarket.com/settings
    │                  (MUST be a contract — not the deposit address!)
    │         signer = Magic-derived EOA (NOT funder)
    │         private_key = exported from https://reveal.magic.link/polymarket
    │
    └─ NO. You're using a stand-alone EOA, never touched polymarket.com UI?
        └─ YES → signature_type = 0 (EOA)
                  funder = signer = your EOA address
                  private_key = your EOA private key
                  ⚠️ rare: most users have a proxy because UI auto-created it
```

**If `funder == signer` it's type 0. If they differ it's type 1 or 2.**
**The UI almost always creates a proxy.** Check polymarket.com/settings — if the displayed address ≠ your wallet's address, you're type 1 or 2.

## Critical: signer ≠ funder for types 1 and 2

| Address | What it is | Where the private key lives | Where pUSD/CTF tokens live |
|---|---|---|---|
| **signer** | EOA that physically signs EIP-712 messages | this is the key you have | usually empty (Metamask key has 0 pUSD) |
| **funder** | proxy contract (POLY_PROXY for type 1, Gnosis Safe for type 2) | not a key — it's a smart contract | here |

**Approvals must be set FROM the funder.** If you `pUSD.approve(exchangeV2, MAX)` from your Metamask EOA but `funder = Gnosis Safe`, the approval is on the wrong account → `INSUFFICIENT_ALLOWANCE` revert at fill time. For type 2 (Gnosis Safe) you must execute approvals **as a Safe transaction** signed by the Safe owner. The polymarket.com UI does this for you when you click "Approve" — for headless bots, you call `Safe.execTransaction(...)` with the owner signature.

## DANGER: deposit address ≠ trading wallet (do NOT confuse them)

**This is the single most common type-2 mistake** in V2. Field report (05.05.2026, similar project): operator put `funder=0x57C5518cdDD704ed44E6817633A3581b1746a5Cb` and got `400 {"error": "maker address not allowed, please use the deposit wallet flow"}` on every order.

Polygonscan check on `0x57C5518c...` showed: **empty EOA, no contract code, $0 balance**. That address was a **deposit address** (ephemeral EOA generated by Polymarket Bridge API to receive USDC.e from external bridges), NOT the trading proxy.

The two are different things:

| | Deposit address | Trading wallet (the real funder) |
|---|---|---|
| Shape | EOA (no contract code) | smart contract |
| Purpose | one-time receiver for an inbound USDC transfer | holds your persistent pUSD + outcome-tokens |
| How obtained | `GET /bridge/deposit-addresses` returns it | derived from your signer EOA via Polymarket Proxy Wallet Factory `0xaB45c5A4B0c941a2F231C04C3f49182e1A254052` (CREATE2) |
| Where it shows | Bridge UI / Bridge API response | polymarket.com/settings → click wallet → Polygonscan shows "Contract" + label |
| Lifetime | ephemeral | permanent (deployed once per EOA) |
| `pUSD.balanceOf(addr)` | always 0 (just a passthrough) | your actual trading balance |

**Litmus test for any candidate `funder`:**

```bash
# On Polygonscan, check the address:
# If it shows "Contract" tab + has bytecode → real proxy/Safe ✓
# If it shows just "Address" with no Contract tab → EOA (probably deposit) ✗
```

Or programmatically:

```python
from web3 import Web3
w3 = Web3(Web3.HTTPProvider('https://polygon-rpc.com'))
code = w3.eth.get_code(Web3.to_checksum_address(candidate_funder))
assert len(code) > 0, (
    f"{candidate_funder} has no contract code — NOT a Safe/proxy, "
    f"probably a deposit address")
```

In plan-kapkan: `python Scripts/poly_verify_funder.py --bot bot1` runs this check + several others.

**To find the REAL trading wallet:**

1. **UI path (recommended)**: log in at polymarket.com → /settings → click on the wallet address shown → opens Polygonscan with that address. The page MUST show "Contract" tab and a label like "Polymarket: Proxy Wallet" or "Gnosis Safe Proxy". That's your `funder`.

2. **SDK path (TypeScript)**:
   ```typescript
   import { deriveSafe } from "@polymarket/builder-relayer-client/dist/builder/derive";
   import { getContractConfig } from "@polymarket/builder-relayer-client/dist/config";
   const safeAddress = deriveSafe(eoaAddress, getContractConfig(137).SafeContracts.SafeFactory);
   ```

3. **Manual CREATE2** (if you really want to skip the SDK): salt + initcode hash from the Polymarket Proxy Wallet Factory contract. Tedious; prefer (1) or (2).

**If polymarket.com/settings shows an EOA-looking address**: you've never connected a wallet through the UI yet, so no Safe is deployed. Connect Metamask via UI ONCE → UI auto-deploys your Safe → /settings updates with the Safe address → use that.

## Two-level auth model (L0 / L1 / L2)

| Level | Mechanism | What it authorizes | Required for |
|---|---|---|---|
| **L0** | nothing | read-only | `GET /markets`, `/book`, `/prices`, gamma-api |
| **L1** | EIP-712 signature with private_key | identity proof for current request | `POST /auth/api-key`, `GET /auth/derive-api-key`, `POST /order` (order body itself signed via L1) |
| **L2** | HMAC-SHA256 with `(api_key, secret, passphrase)` | session-style request auth | `DELETE /order/{id}`, `DELETE /orders`, `GET /data/positions`, user-channel WS |

**`POST /order` requires BOTH L1 and L2:** the order struct is L1-signed inside the body, and the request itself ALSO carries L2 HMAC headers in V2.

### How L2 creds are derived from L1 — and the (signer, funder) binding

```
1. Build EIP-712 ClobAuth message with private_key
2. POST /auth/api-key (or GET /auth/derive-api-key for re-fetch)
   Headers: POLY_ADDRESS (signer), POLY_SIGNATURE, POLY_TIMESTAMP, POLY_NONCE
3. Server returns { apiKey, secret, passphrase }
   THESE CREDS ARE BOUND TO (signer, funder, sig_type) TUPLE
4. Reuse on every request
```

**Critical pitfall (field report)**: deriving creds in `signature_type=0` mode (no funder passed) and then trading in `signature_type=2` mode (with funder) → server treats them as different accounts → `401` or `"maker address not allowed"`. The fix: pass `signature_type` and `funder` to the temp client used for derivation:

```python
# WRONG — derives creds in EOA mode
temp = ClobClient(host, chain_id, key=PK)
creds = temp.create_or_derive_api_key()

# RIGHT — derives creds in same mode you'll trade in
temp = ClobClient(host, chain_id, key=PK,
                  signature_type=2, funder=SAFE_ADDRESS)
creds = temp.create_or_derive_api_key()
```

The signed ClobAuth message:
```python
domain = {"name": "ClobAuthDomain", "version": "1", "chainId": 137}
types  = {"ClobAuth": [
    {"name": "address",   "type": "address"},
    {"name": "timestamp", "type": "string"},
    {"name": "nonce",     "type": "uint256"},
    {"name": "message",   "type": "string"},
]}
value  = {
    "address":   signer_address,    # NOT funder
    "timestamp": str(int(time.time())),
    "nonce":     0,
    "message":   "This message attests that I control the given wallet",
}
```

**Note:** ClobAuthDomain version stays `"1"` even in V2. Only the **Order** EIP-712 domain went to version `"2"`. Don't confuse them.

## Package selection — CRITICAL

| Package | Version | Status | Use? |
|---|---|---|---|
| `py-clob-client` | ≤ 0.34.6 (latest is 19 Feb 2025) | **V1 only — broken on V2** (hardcodes `CLOB_VERSION = "1"` in `eip712.py` → server returns `order_version_mismatch`) | **NO** |
| **`py-clob-client-v2`** | **≥ 1.0.0 (released 17.04.2026)** | Official V2 package | **YES** |
| `@polymarket/clob-client` (TS V1) | latest | broken on V2 | **NO** |
| **`@polymarket/clob-client-v2`** (TS V2) | latest | Official V2 TS package | **YES** |

```bash
# WRONG (will fail with order_version_mismatch on every order):
pip install py-clob-client

# RIGHT:
pip install py-clob-client-v2
```

**If you wrote your own signer with `requests` and EIP-712**, make sure the Order domain has `version: "2"` and the V2 verifyingContract addresses (see `polymarket-v2-connector`).

## Init templates

### Type 0 (pure EOA — rare)
```python
from py_clob_client_v2.client import ClobClient

client = ClobClient(
    host="https://clob.polymarket.com",
    chain_id=137,
    key=PRIVATE_KEY,
    # no signature_type, no funder — defaults to type 0 with funder=signer
)
creds = client.create_or_derive_api_creds()
client.set_api_creds(creds)
```

### Type 1 (Magic email — exported key)
```python
client = ClobClient(
    host="https://clob.polymarket.com",
    chain_id=137,
    key=MAGIC_EXPORTED_PRIVATE_KEY,
    signature_type=1,
    funder=POLYMARKET_PROXY_ADDRESS,  # from polymarket.com/settings (CONTRACT)
)
# Derive creds in same mode — bound to (signer, funder)
creds = client.create_or_derive_api_creds()
client.set_api_creds(creds)
```

### Type 2 (Browser wallet — Gnosis Safe via polymarket.com UI)
```python
client = ClobClient(
    host="https://clob.polymarket.com",
    chain_id=137,
    key=METAMASK_PRIVATE_KEY,
    signature_type=2,
    funder=GNOSIS_SAFE_ADDRESS,  # from polymarket.com/settings (CONTRACT)
)
# Derive creds in same mode — bound to (signer, funder)
creds = client.create_or_derive_api_creds()
client.set_api_creds(creds)
```

## How to verify your setup is correct (smoke checks)

Run these in order. Any failure = the step that follows is undefined.

```python
# 1. Signer key matches the EOA you think it does
from eth_account import Account
acct = Account.from_key(PRIVATE_KEY)
print("signer:", acct.address)  # must match polymarket.com displayed wallet
                                # for type 0; or the "Magic" address for type 1;
                                # or your Metamask connected address for type 2

# 2. Funder is a CONTRACT (rules out the deposit-address mistake)
import requests
from web3 import Web3
w3 = Web3(Web3.HTTPProvider('https://polygon-rpc.com'))
assert len(w3.eth.get_code(Web3.to_checksum_address(FUNDER))) > 0, \
    f"FUNDER {FUNDER} has no contract code — that's an EOA, not a proxy/Safe"

# 3. Funder has pUSD balance
pusd = "0xC011a7E12a19f7B1f670d46F03B03f3342E82DFB"
bal = call_erc20_balance(pusd, FUNDER)
assert bal > 0, f"Funder {FUNDER} has 0 pUSD — wrap USDC.e first"

# 4. pUSD allowance to V2 Exchange
exchange_std = "0xE111180000d2663C0091e4f400237545B87B996B"
allow = call_erc20_allowance(pusd, FUNDER, exchange_std)
assert allow > 10**24, "Run polymarket_approve.py — pUSD not approved"

# 5. L1 derives creds successfully (in matching sig_type/funder mode!)
creds = client.create_or_derive_api_creds()
assert creds.api_key and creds.api_secret and creds.api_passphrase

# 6. L2 read works (positions endpoint requires HMAC)
positions = client.get_positions()  # should return [] not raise 401
```

In plan-kapkan: `python Scripts/poly_verify_funder.py --bot bot1` runs checks 1-4.

## Common confusion to clear up

> "Я зарегистрировался И через метамаск И через почту — это разные аккаунты"

**Yes — they are completely different accounts with different funder addresses, different proxies, different pUSD balances, different API creds.** Pick ONE. Don't mix keys/funders across the two.

> "API creds должны быть связаны с private key и funder"

Yes — and **HOW you derive them matters**. If you call `create_or_derive_api_key()` from a client initialized in EOA mode (no `signature_type`/`funder`), creds bind to `(signer, EOA)`. Then if you trade with `signature_type=2, funder=Safe`, server says "creds belong to a different account" → 401. Always derive in the SAME mode you'll trade in.

> "Видел signature_type=3 / NONE"

**Doesn't exist.** Only 0, 1, 2. Source: `py-clob-client-v2` v1.0.0 and docs.polymarket.com/api-reference/authentication.

> "L1 проходит — значит signer/funder правильные"

**No.** L1 only proves you control the private_key for `signer`. It does NOT verify funder is correct. You can derive valid API creds with `funder=0x000...000` and still fail at `POST /order` with `INVALID_SIGNATURE` because the order body's signer/maker/funder triple doesn't recover correctly.

> "Я нашёл funder адрес у себя в Bridge UI / при депозите"

**That's a deposit address, NOT the trading wallet.** Deposit addresses are ephemeral EOAs that the Bridge API generates to receive USDC; they do NOT hold your pUSD. The real trading wallet is a SMART CONTRACT shown at polymarket.com/settings. See "DANGER" section above.

## SOCKS5 routing for L2 endpoints (per-bot static IP)

**Why:** Polymarket runs Cloudflare. Cloudflare fingerprints requests by `(IP, headers, cookies, JA3)` and flags accounts whose IP geo "wanders" mid-session — flag → AML review → ban. With multiple bots on one machine, all 6 bots also share the host's outbound IP by default → single IP for 6 distinct (signer, funder) tuples → Cloudflare sees one machine running multiple wallets → flag.

**Solution:** route only L2-authenticated calls through a SOCKS5 proxy where each bot pins a static port → static exit IP.

### What goes through proxy vs direct

| Layer | Endpoints | Proxy? |
|---|---|---|
| L0 public | `/book`, `/markets`, `/prices`, gamma-api | direct |
| L1 derive | `POST /auth/api-key`, `GET /auth/derive-api-key` | direct |
| **L2 auth** | `POST /order`, `DELETE /order/{id}`, `DELETE /orders`, `GET /data/positions`, user-channel WS | **SOCKS5** |

L0/L1 are not account-attributed (no api_key in request) → IP doesn't matter for fingerprinting.
L2 carries `POLY_API_KEY` → server attributes the request to the wallet → IP-geo coherence matters.

### Port-per-bot static mapping

The radar maps `bot{N}` → port `POLY_PROXY_PORT_BASE + (N-1)`. Default base 10000:

| bot | port | exit IP |
|---|---|---|
| bot1 | 10000 | static IP A |
| bot2 | 10001 | static IP B |
| ... | ... | ... |
| bot6 | 10005 | static IP F |

Each bot keeps the **same** exit IP across cancels, orders, position polls, WS reconnects. This is the right tradeoff: anti-fingerprint comes from IP + (signer, funder) being **consistent**, not from IP rotation.

### Wiring in plan-kapkan

- `Scripts/poly_l2_http.py` — single source of truth. `l2_request(method, path, wallet=...)` builds HMAC headers + sends via the bot's pinned proxy session.
- `Scripts/poly_user_ws.py` — user-channel WS via SOCKS5 (auth-bearing connection)
- `executor/atomic.py`, `watchdog.py`, `risk/reconcile.py`, `arb_server.py` — all migrated to `l2_request`
- `Scripts/poly_proxy_check.py` — pre-flight: per-bot exit IP + collision check

### Configuration

In `Credentials.env`:
```env
POLY_PROXY_HOST=pool.proxy.market
POLY_PROXY_USER=<provider-issued user>
POLY_PROXY_PASS=<provider-issued password>
POLY_PROXY_PORT_BASE=10000
POLY_PROXY_PORT_END=10999
```

Leave blank for direct mode (dev/paper). Half-set (HOST without USER/PASS) hard-fails — refusing to silently leak the bot's real IP if the operator intended proxy but typo'd creds.

### Verification

```bash
python Scripts/poly_proxy_check.py --geo
```

Expected output:
```
bot    port   exit_ip            country  status
--------------------------------------------------------
bot1   10000  185.X.Y.Z         GE       ✅
bot2   10001  185.X.Y.W         GE       ✅
...
✅ All 6 bot(s) routed correctly with distinct exit IPs.
```

Failure modes the check catches:
- PySocks missing → install hint
- Wrong creds → 407 from proxy
- Two bots got same IP (port collision)
- IP geo doesn't match `ALLOWED_COUNTRIES`
